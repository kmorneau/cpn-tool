The cpn Text DSL

A plain-text language for defining Colored Petri Nets. Write nets as readable source, simulate them live, and export to PNML for use with other tools.

CPN Tool · Professional Edition · DSL v1.0

Introduction

The cpn DSL is a line-oriented declarative language. Each line is one declaration. Order within the file is irrelevant to semantics — places, transitions, and arcs can appear in any sequence — though the conventional order shown throughout this document improves readability.

A net definition consists of four kinds of declaration: colorsets, places, transitions, and arcs. Together these define the complete structure and initial state of a Colored Petri Net.

Note — Relationship to CPN Theory

This DSL maps directly onto the formal CPN model: color sets are token types, places are typed token holders, transitions are guarded firing rules, and arcs carry inscription expressions that determine which tokens are consumed and produced. The simulation engine performs binding-based enabling and firing as defined in the CPN formalism.

File Structure

A .cpn file is plain UTF-8 text. The recommended structure has four sections — colorsets first, then places, then transitions, then arcs — separated by blank lines. This order is a convention, not a requirement.

Recommended file layout
// Section 1: Color set declarations
colorset JOBS    = {job1, job2, job3}
colorset WORKERS = {w1, w2}

// Section 2: Place declarations
place Ready : JOBS    = {job1, job2, job3}
place Idle  : WORKERS = {w1, w2}
place Busy  : JOBS*WORKERS = {}
place Done  : JOBS    = {}

// Section 3: Transition declarations
transition Assign   [guard: j in JOBS, w in WORKERS]
transition Complete

// Section 4: Arc declarations
arc Ready    --> Assign   : j
arc Idle     --> Assign   : w
arc Assign   --> Busy     : (j,w)
arc Busy     --> Complete : (j,w)
arc Complete --> Done     : j
arc Complete --> Idle     : w

Grammar Summary

The grammar is line-oriented. Each non-blank, non-comment line must match exactly one of the four declaration forms. Leading and trailing whitespace is stripped before matching.

program ::= (line '\n')* line ::= comment | blank | colorset-decl | place-decl | transition-decl | arc-decl comment ::= ('//' | '#') any-text blank ::= whitespace-only colorset-decl ::= 'colorset' NAME '=' '{' value-list '}' place-decl ::= 'place' NAME ':' COLORSET-TYPE '=' '{' value-list '}' transition-decl ::= 'transition' NAME ['[guard:' guard-text ']'] arc-decl ::= 'arc' NAME '-->' NAME ':' arc-expr NAME ::= word // \w+ : letters, digits, underscore COLORSET-TYPE ::= NAME ('*' | '×') NAME // product type | NAME // simple type value-list ::= (empty) | NAME (',' NAME)* arc-expr ::= NAME // variable or literal | '(' NAME ',' NAME ')' // pair tuple guard-text ::= any-text-until-']' // stored as-is, displayed on canvas

colorset

Declares a named color set — the type of tokens that places may hold. A color set is an enumeration of token values. Tokens in the net must come from their place's declared color set.

Syntax

Syntax
colorset NAME = { value1, value2, ... }
NAME required
The color set identifier. Must be a valid name (\w+). By convention, use ALL_CAPS to distinguish color set names from place and transition names.
{ value1, value2, ... } required
A comma-separated list of token values enclosed in curly braces. Each value is a word (\w+). The list may be empty ({}), though an empty color set produces a place that can never hold tokens.

Examples

colorset JOBS    = {job1, job2, job3}    // three job tokens
colorset WORKERS = {w1, w2}              // two worker tokens
colorset SEM     = {s}                  // single semaphore token
colorset DATA    = {a, b, c}            // generic data tokens
Convention

Color set declarations are optional in the parser — you can use undeclared names as colorset types in place declarations. However, declaring them gives you a visible legend in the tool's analysis panel and makes the net self-documenting.

place

Declares a place in the net. A place is a typed container that holds a multiset of tokens. Every place has a declared color set type and an initial marking — the set of tokens present when the simulation starts.

Syntax

Syntax
place NAME : COLORSET = { token1, token2, ... }
NAME required
The place identifier. Used in arc declarations to reference this place. Must be unique across all places and transitions in the net.
: COLORSET required
Declares the color set type of tokens this place may hold. Three forms are accepted:
  • JOBS — simple named type
  • JOBS*WORKERS — product type using *
  • JOBS×WORKERS — product type using × (U+00D7)
Product types indicate the place holds tuple tokens of the form (val1,val2).
= { token1, token2, ... } required
The initial marking — the tokens present at the start of simulation. Token values must match the declared color set. Use {} for an initially empty place. Repeated tokens are allowed and create a multiset (e.g., {a, a, b} = two a tokens plus one b).

Examples

// Simple color set: place holds JOBS tokens, starts with 3
place Ready    : JOBS         = {job1, job2, job3}

// Initially empty place (tokens arrive via arc firing)
place Done     : JOBS         = {}

// Product type: holds tuples of (JOBS, WORKERS)
place Busy     : JOBS*WORKERS = {}

// Unicode × is also accepted
place Eating   : PHIL×FORK    = {}

// Semaphore: one token of value 's'
place Semaphore : SEM         = {s}

// Multiset initial marking: two identical tokens
place Buffer   : DATA         = {a, a, b}
Simulation constraint

Initial tokens in the { } list should be values that appear in the declared color set. The parser does not type-check tokens against color set definitions — mismatches produce no error at parse time but may cause transitions to never fire if bindings cannot be satisfied.

transition

Declares a transition in the net. Transitions are the active elements of a CPN — they fire when enabled, consuming tokens from input places and producing tokens on output places. Firing is governed by a binding of variables to token values, subject to the guard condition.

Syntax

Syntax (two forms)
// Without guard (always enabled when input tokens available)
transition NAME

// With guard (additional enabling condition)
transition NAME [guard: guard-expression]
NAME required
The transition identifier. Used in arc declarations. Must be unique across all places and transitions.
[guard: expression] optional
A guard condition enclosed in square brackets, prefixed with guard:. The guard text is stored as-is and displayed on the canvas — the parser does not evaluate it. A transition without a guard is treated as having guard true (always enabled when input arcs are satisfied). A transition with [guard: true] is equivalent to having no guard.

Examples

// No guard — fires whenever input places have tokens
transition Complete

// Guard with variable membership assertions (documentation)
transition Assign   [guard: j in JOBS, w in WORKERS]

// Explicit true guard (same as no guard)
transition Pop      [guard: true]

// Guard can contain any descriptive text
transition Enter    [guard: p in PROC, semaphore available]
Guard semantics in this DSL

Guard expressions in the current DSL are descriptive, not evaluated. The simulation engine enables a transition based solely on whether all input arc expressions can be satisfied by available tokens. The guard text is shown on the canvas as a label for human understanding. A future version may evaluate boolean guard expressions against the active binding.

arc

Declares a directed arc connecting a place to a transition (input arc) or a transition to a place (output arc). Each arc carries an expression that determines which tokens are consumed or produced when the transition fires. Arcs enforce the net's structure — an arc must connect a place to a transition or a transition to a place; place-to-place and transition-to-transition connections are not permitted.

Syntax

Syntax
arc SOURCE --> TARGET : expression
SOURCE required
The name of the source node — either a place name or a transition name.
--> required
The arc direction operator. Always points from source to target.
TARGET required
The name of the target node. Must be the opposite kind from source: if source is a place, target must be a transition, and vice versa.
: expression required
The arc inscription — an expression evaluated against the current binding during firing. See Arc Expressions for the full expression language.

Arc direction and role

Arc directionRoleEffect when transition fires
place --> transition Input arc Consumes one token from the place matching the expression value. The transition cannot fire if the place lacks a matching token.
transition --> place Output arc Produces one token onto the place equal to the expression value resolved against the current binding.

Examples

// Input arc: consume one j-token from Ready when Assign fires
arc Ready    --> Assign   : j

// Input arc: consume one w-token from Idle
arc Idle     --> Assign   : w

// Output arc: produce a tuple (j,w) onto Busy
arc Assign   --> Busy     : (j,w)

// Input arc consuming a tuple: binds both j and w from the token
arc Busy     --> Complete : (j,w)

// Output arc with literal: always produces the same token value
arc Exit     --> Semaphore : s

Arc Expressions

The arc expression language has three forms. Every arc must use exactly one form per arc — expressions cannot be combined or nested beyond what is shown here.

Form 1 — Variable

x

A single identifier. When used on an input arc, it acts as a variable: the binding engine assigns the variable to one available token in the source place. When used on an output arc, it resolves to the value the variable was bound to during this firing.

ContextBehaviourExample
Input arcBinds variable to one token from the placearc Ready --> Assign : j — binds j to e.g. job1
Output arcProduces the token that variable is bound toarc Complete --> Done : j — produces job1

Form 2 — Tuple

(x,y)

A pair of two variable names enclosed in parentheses, separated by a comma. Used with product-type places whose tokens are of the form (val1,val2).

ContextBehaviourExample
Input arc from product placeDestructures a tuple token, binding each part to its variablearc Busy --> Complete : (j,w) — if Busy contains (job1,w1), binds j=job1 and w=w1
Output arc to product placeConstructs a tuple from the two bound variable valuesarc Assign --> Busy : (j,w) — produces (job1,w1)
Unbound variable in tupleThe unbound name is used as a literal(j,y) where only j is bound → produces (job1,y)

Form 3 — Literal

s

A word that is not bound as a variable. When the binding engine cannot find a match for this name in the current binding, it is used as a literal token value. On input arcs, it requires that the exact token value s be present in the source place. On output arcs, it always produces that exact token.

ContextBehaviourExample
Input arcPlace must contain the literal token valuearc Semaphore --> Enter : s — requires token s in Semaphore
Output arcAlways produces the literal tokenarc Exit --> Semaphore : s — always puts token s back
Variable vs literal — how the engine decides

The binding engine resolves expressions at firing time. On an input arc, a single-word expression is treated as a variable to bind. On an output arc, a single word is looked up in the current binding; if found, the bound value is used; if not found, the word is used as a literal. The same name can act as a variable on input arcs and resolve to its bound value on output arcs.

Guard Expressions

A guard is free-form text written after guard: inside square brackets. The current version of the DSL treats guards as documentation — they are parsed, stored, and displayed on the canvas alongside the transition, but they are not evaluated during simulation. A transition is enabled based on arc satisfaction alone.

Guard syntax
transition NAME [guard: any text until the closing bracket]

Conventional guard patterns

Though not evaluated, guards are most useful when they follow conventions that make the net's intent clear to readers.

PatternMeaningExample
x in CSVariable must belong to color setj in JOBS
x in CS, y in CS2Multiple membership conditionsj in JOBS, w in WORKERS
trueUnconditional (same as no guard)true
x < yOrdering condition (descriptive)priority(j) > priority(k)
Free textAny prose descriptionsemaphore not held

Variable Binding

When the simulation engine checks whether a transition is enabled, it attempts to find a binding — an assignment of variable names to token values — such that all input arcs can be simultaneously satisfied.

How binding works

The engine processes the transition's input arcs left-to-right (in the order arcs appear in the file). For each input arc:

  1. If the arc expression is a single variable (e.g. j), try each available token in the source place as a candidate binding for j.
  2. If the expression is a tuple (x,y), try each available token of the form (v1,v2), binding x=v1 and y=v2.
  3. If the expression is a literal, require that exact token value to be present.
  4. Continue to the next arc with the extended binding. If a later arc cannot be satisfied with the current binding, backtrack and try the next candidate.

If a complete binding satisfying all input arcs is found, the transition is enabled. Multiple valid bindings may exist — the simulation engine selects one (the first found, which depends on token order in the marking).

Binding example — step by step

// Net: Assign has two input arcs
arc Ready  --> Assign : j   // arc 1: bind j
arc Idle   --> Assign : w   // arc 2: bind w

// Marking: Ready = {job1, job2}, Idle = {w1, w2}
//
// Binding attempt:
//   Arc 1: try j = job1  →  binding = {j: job1}
//   Arc 2: try w = w1    →  binding = {j: job1, w: w1}  ✓ all arcs satisfied
//
// Firing with binding {j: job1, w: w1}:
//   Consume job1 from Ready  →  Ready = {job2}
//   Consume w1 from Idle     →  Idle = {w2}
//   arc Assign --> Busy : (j,w)  →  produce (job1,w1) on Busy

Identifiers

All names in the DSL — color set names, place names, transition names, and token values — follow the same rule: they are words matching the pattern \w+, meaning one or more word characters: letters (a–z, A–Z), digits (0–9), or underscores (_). No spaces, hyphens, or special characters are permitted within a name.

Identifier typeConventionExamples
Color set namesALL_CAPSJOBS, WORKERS, DATA, SEM
Place namesPascalCase or plain wordReady, Idle, Semaphore, Buffer
Transition namesPascalCase or verbAssign, PickUp, Enter, Push
Token valuesShort lowercasejob1, w1, s, p1, f1
Arc variablesSingle lowercase letterj, w, x, p, f
Name uniqueness

Place names and transition names share the same namespace. A name used for a place cannot be used for a transition in the same net. Arc declarations reference nodes by name — if a name is ambiguous, the first match (places searched before transitions) is used.

Comments and Whitespace

Line comments

Lines beginning with // or # (after optional leading whitespace) are treated as comments and ignored entirely.

// This is a comment
// Multiple comment lines are fine

# Hash-style comments also work
# Useful for scripting interoperability

colorset DATA = {a, b}    // inline comment NOT supported — full-line only
Inline comments not supported

Comments must occupy a full line. There is no inline comment syntax — anything after a declaration on the same line is not parsed as a comment and will cause a parse error if it doesn't match the declaration pattern. Write comments on their own lines.

Blank lines

Blank lines (including lines containing only whitespace) are ignored. Use them freely to separate sections and improve readability.

Leading and trailing whitespace

Each line is trimmed of leading and trailing whitespace before parsing. Indentation has no semantic meaning. You may align declarations with spaces for visual clarity — this is encouraged.

// These are identical to the parser:
place Ready : JOBS = {job1}
    place Ready : JOBS = {job1}

// Alignment is cosmetic only, but improves readability:
place Ready     : JOBS         = {job1, job2, job3}
place Idle      : WORKERS      = {w1, w2}
place Semaphore : SEM          = {s}

Parse Errors

Any non-blank, non-comment line that does not match one of the four declaration patterns is recorded as a parse error. Errors are reported with the line number and the unrecognized text. The parser continues after an error — subsequent valid lines are still processed. A net with parse errors is still simulated using whatever valid declarations were successfully parsed.

Common error causeExample bad lineFix
Misspelled keyword colour JOBS = {a} Use colorset
Missing colon in place place Ready JOBS = {a} place Ready : JOBS = {a}
Missing arrow in arc arc Ready Assign : j arc Ready --> Assign : j
Missing colon after target in arc arc Ready --> Assign j arc Ready --> Assign : j
Inline comment place Ready : JOBS = {} -- empty Move comment to its own line
Space in name place My Place : JOBS = {} place MyPlace : JOBS = {}
Missing equals sign colorset JOBS {a, b} colorset JOBS = {a, b}
Live error feedback

In the Text editor mode, errors appear in the red error bar below the editor as you type, with the line number highlighted. The status indicator at the top right shows a red dot when errors are present and a green dot when the net parses cleanly.

Complete Examples

Producer-Consumer
colorset JOBS    = {job1, job2, job3}
colorset WORKERS = {w1, w2}

place Ready : JOBS         = {job1, job2, job3}
place Idle  : WORKERS      = {w1, w2}
place Busy  : JOBS*WORKERS = {}
place Done  : JOBS         = {}

transition Assign   [guard: j in JOBS, w in WORKERS]
transition Complete

arc Ready    --> Assign   : j
arc Idle     --> Assign   : w
arc Assign   --> Busy     : (j,w)
arc Busy     --> Complete : (j,w)
arc Complete --> Done     : j
arc Complete --> Idle     : w
Mutual Exclusion
colorset PROC = {p1, p2}
colorset SEM  = {s}

place Ready     : PROC = {p1, p2}
place Critical  : PROC = {}
place Done      : PROC = {}
place Semaphore : SEM  = {s}

transition Enter [guard: p in PROC]
transition Exit

arc Ready     --> Enter     : p
arc Semaphore --> Enter     : s
arc Enter     --> Critical  : p
arc Critical  --> Exit      : p
arc Exit      --> Done      : p
arc Exit      --> Semaphore : s
Dining Philosophers
colorset PHIL = {p1, p2}
colorset FORK = {f1, f2}

place Thinking : PHIL      = {p1, p2}
place Eating   : PHIL*FORK = {}
place Forks    : FORK      = {f1, f2}

transition PickUp  [guard: p in PHIL, f in FORK]
transition PutDown

arc Thinking --> PickUp  : p
arc Forks    --> PickUp  : f
arc PickUp   --> Eating  : (p,f)
arc Eating   --> PutDown : (p,f)
arc PutDown  --> Thinking: p
arc PutDown  --> Forks   : f
Token Pipeline
colorset DATA = {a, b, c}

place Input  : DATA = {a, b, c}
place Buffer : DATA = {}
place Output : DATA = {}

transition Push
transition Pop

arc Input  --> Push   : x
arc Push   --> Buffer : x
arc Buffer --> Pop    : x
arc Pop    --> Output : x
